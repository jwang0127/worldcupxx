世界杯实时数据 API 评估

目标是解决两个问题：能不能实时更新赛果，以及能不能持续拿到下一场世界杯比赛进程。结论是可以，但建议“主源 + 备源 + 兜底源”三层校验，不要只依赖单一接口。

推荐方案

主源：API-Football 备源：football-data.org 兜底：SerpApi

实时赛程、开赛状态、比分、完场结果优先由 API-Football 处理；football-data.org 做赛程和结果核验；SerpApi 抓 Google 体育卡片做异常兜底。

自动结算规则

两源一致自动结算 冲突进入人工确认 赔率与赛果分仓

当主源和备源的全场比分、比赛状态都一致时写入 `result`；如果比分或状态冲突，页面保留“待回填”，避免误结算污染模型复盘。

数据源 能否更新赛果 能否跟踪下一场进程 适合用途 风险与备注
API-Football 可以 可以 Fixtures、Live status、elapsed、events、lineups、statistics，最适合作为实时主源。 需要 API Key；世界杯覆盖要按 plan 和 competition/league id 实测确认。
football-data.org 可以 偏结果与赛程 赛程、比分、积分榜、结果复核，适合作为结算备源。 实时粒度通常不如 API-Football，免费层有限额。
SerpApi Google 可以兜底 取决于 Google 卡片 当官方 API 失败时，抓 Google 搜索体育卡片里的比分/状态。 结构不是专用体育 API，字段稳定性弱，不建议做唯一结算源。
TheSportsDB 可做补充 LiveScore 通常需高级权限 球队、球场、赛事资料、赛程镜像、历史信息。 免费版更像资料库；实时性和覆盖需按接口权限验证。

建议接入闭环

1. 拉赛程每日 13:00 读取下一比赛日列表。
2. 拉赔率Sporttery 或截图 OCR 写入总进球赔率。
3. 生成预测输出每日四场 HTML 看板。
4. 拉赛果完场后主源/备源交叉校验。
5. 复盘更新结算命中率、更新模型权重并推送网页。
